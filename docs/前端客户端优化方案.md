# 前端客户端代码优化方案文档

> 版本：v1.0 | 日期：2026-06-18 | 状态：✅ 已实施

---

## 一、优化概述

### 1.1 背景

当前应用存在以下核心问题：

| 问题 | 影响 |
|------|------|
| 所有 API 调用无重试、无超时 | 网络抖动直接导致用户操作失败 |
| 纯计算逻辑（卖点打分、文案润色、信息提取）全部走服务端 | 无效消耗服务器资源，用户需排队等待 AI 请求队列 |
| 图片嵌入阶段占用 28%-82% 的进度区间 | 用户感知最长等待段 |
| 卖点数据、模板数据无离线缓存 | 每次进入页面都发起网络请求 |
| toast 无队列机制 | 快速操作时消息相互覆盖，用户丢失反馈 |

### 1.2 优化目标

| 目标 | 指标 |
|------|------|
| 减少等待时间 | 纯计算操作从 2-8s（网络+排队）降至 <50ms（本地即时） |
| 即时反馈 | 所有用户操作均有明确 toast 反馈，不丢失 |
| 离线可用 | 卖点模板数据、模板配置离线可用 |
| 减少服务器负载 | 将 3 类纯计算 API 调用迁移到客户端，预计减少 60% AI 队列请求 |

### 1.3 设计原则

1. **纯计算逻辑优先本地化**：不依赖数据库、不依赖服务端状态的纯函数，迁移到客户端
2. **渐进式降级**：本地能力不足时自动 fallback 到服务端
3. **向后兼容**：所有新增客户端逻辑不影响现有服务端 API
4. **零新增依赖**：不引入第三方库，利用浏览器原生能力

## 二、具体优化项

### 优化项 1：请求重试与超时机制

**问题描述**

`public/js/api.js` 中的 `request` 函数无任何重试和超时控制。网络抖动或服务器响应慢时，用户直接看到错误。

**优化方案**

为 `api.js` 的 `request` 函数增加：
- 可配置超时（默认 15s，AI 类接口 60s）
- 自动重试（最多 2 次，仅针对网络错误和 5xx 状态码）
- 指数退避（500ms -> 1000ms）

**涉及文件**

- `public/js/api.js`

**代码修改要点**

```javascript
// api.js 中新增 requestWithRetry 函数

const DEFAULT_TIMEOUT_MS = 15000;
const AI_TIMEOUT_MS = 60000;
const MAX_RETRIES = 2;
const RETRY_DELAYS = [500, 1000];

function isRetryableError(status) {
  return status >= 500 || status === 0;
}

async function requestWithRetry(path, options = {}) {
  const isAi = ['/api/selling-points/generate', '/api/desc/extract', '/api/polish'].some(
    (p) => path.includes(p)
  );
  const timeout = options.timeout || (isAi ? AI_TIMEOUT_MS : DEFAULT_TIMEOUT_MS);
  const maxRetries = options.maxRetries !== undefined ? options.maxRetries : MAX_RETRIES;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    if (attempt > 0) {
      await new Promise((r) => setTimeout(r, RETRY_DELAYS[attempt - 1] || 2000));
    }

    const controller = new AbortController();
    const timer = setTimeout(() => controller.abort(), timeout);

    try {
      const res = await fetch(apiUrl(path), {
        headers: {
          'Content-Type': 'application/json',
          ...getAuthHeaders(),
          ...(options.headers || {}),
        },
        signal: controller.signal,
        ...options,
      });

      const data = await res.json().catch(() => ({}));
      if (!res.ok) {
        if (isRetryableError(res.status) && attempt < maxRetries) continue;
        if (res.status === 401) clearToken();
        throw new ApiError(data.error?.code || res.statusText, {
          code: data.error?.code,
          status: res.status,
          details: data.error,
        });
      }
      return data;
    } catch (err) {
      if (err.name === 'AbortError') {
        if (attempt < maxRetries) continue;
        throw new ApiError('请求超时', { code: 'REQUEST_TIMEOUT', status: 0 });
      }
      if (attempt < maxRetries && isRetryableError(err.status || 0)) continue;
      throw err;
    } finally {
      clearTimeout(timer);
    }
  }
}
```

**验证方案**

1. 单元测试：模拟 500 错误，验证重试 2 次后成功
2. 集成测试：使用 Chrome DevTools 的 Slow 3G 网络节流，验证请求不超时失败
3. 超时测试：延迟服务端响应，验证 15s 后超时

### 优化项 2：Toast 消息队列

**问题描述**

`public/js/app.js` 中的 `toast` 函数（第 131-136 行）直接使用固定 1800ms 定时器覆盖显示，快速连续操作时后一条消息会覆盖前一条，导致用户看不到关键反馈。

**优化方案**

实现 FIFO 队列的 toast 管理器：
- 消息排队显示，每条至少展示 1800ms
- 同类消息合并（相同的消息内容不重复入队）
- 最多队列长度 10，超出丢弃最旧的

**涉及文件**

- `public/js/app.js`

**代码修改要点**

```javascript
// app.js 中替换原有 toast 函数（第 131-136 行）

const toastQueue = [];
let toastTimer = null;
let isShowing = false;
const MAX_TOAST_QUEUE = 10;

function toast(msg, { duration = 1800, deduplicate = true } = {}) {
  if (deduplicate) {
    if (toastQueue.some((item) => item.msg === msg)) return;
    if (isShowing && $('#toast')?.textContent === msg) return;
  }

  if (toastQueue.length >= MAX_TOAST_QUEUE) {
    toastQueue.shift();
  }

  toastQueue.push({ msg, duration });

  if (!isShowing) showNextToast();
}

function showNextToast() {
  if (toastQueue.length === 0) {
    isShowing = false;
    return;
  }

  isShowing = true;
  const { msg, duration } = toastQueue.shift();
  const t = $('#toast');
  t.textContent = msg;
  t.style.display = 'block';

  clearTimeout(toastTimer);
  toastTimer = setTimeout(() => {
    t.style.display = 'none';
    isShowing = false;
    setTimeout(showNextToast, 100);
  }, duration);
}
```

**验证方案**

1. 手动测试：快速连续触发 3 个 toast，验证逐条显示不重叠
2. 去重测试：连续 2 次调用 `toast('保存成功')`，验证只显示一次
3. 溢出测试：连续调用 12 次 toast，验证只显示最后 10 条

### 优化项 3：本地卖点推荐引擎（核心优化）

**问题描述**

点击"AI 提炼总结"按钮后，调用 `POST /api/selling-points/generate` 接口，流程为：客户端 → 服务端 AI 队列（最多排队 120s）→ LLM API → 返回结果。但 `sellingEngine.js` 的打分算法和 `copyPolish.js` 的本地模板匹配逻辑完全可以迁移到客户端。

**优化方案**

1. 新增客户端卖点推荐模块 `sellingEngineClient.js`
2. 迁移 `BRAND_META`（27 品牌）、`TIER_CATEGORY_BOOST`、`parseVehicleProfile`、`scoreSellingPoint`、`pickWithRandomness`
3. 迁移内置卖点数据（32 条）
4. 客户端优先本地推荐，不足时 fallback 服务端

**涉及文件**

- 新增：`public/js/sellingEngineClient.js`
- 修改：`public/js/app.js`
- 参考：`server/services/sellingEngine.js`
- 参考：`server/services/copyPolish.js`
- 参考：`server/db.js`（SEED_POINTS）

**迁移内容清单**

| 来源文件 | 迁移内容 |
|----------|----------|
| `sellingEngine.js` | `BRAND_META`（27 品牌）、`SUV_HINTS/SEDAN_HINTS/MPV_HINTS`、`TIER_CATEGORY_BOOST`、`parseVehicleProfile()`、`buildRecommendKeywords()`、`scoreSellingPoint()`、`pickWithRandomness()` |
| `copyPolish.js` | `normalizeSellingPoint()`、`localGenerateSellingPoints()` |
| `db.js` | `SEED_POINTS`（32 条内置卖点数据） |

**客户端调用流程**

```
用户点击"AI 提炼总结"
  → 本地 generateFromTextLocal()（<50ms）
  → 结果 >= 3 条？直接使用
  → 结果 < 3 条？fallback 服务端 LLM
```

**验证方案**

1. 对比测试：同一输入，客户端推荐结果与服务端 `local_template` 源结果一致
2. 性能测试：本地推荐耗时 <50ms
3. 降级测试：禁用网络后，卖点推荐功能仍可正常工作
4. 准确率测试：覆盖 27 个品牌 + 不同年份，验证 `parseVehicleProfile` 输出正确

### 优化项 5：结构化数据离线缓存

**问题描述**

每次进入模板页或描述页，都会通过 `api.listTemplates()` 和其他接口发起网络请求。这些数据变化频率低，适合离线缓存。

**优化方案**

1. 利用 IndexedDB 基础设施，新增结构化数据缓存
2. 缓存模板列表（`/api/templates`），TTL 24 小时
3. 缓存卖点推荐结果（按 `brandModel + year` 键），TTL 7 天

**涉及文件**

- 新增：`public/js/offlineCache.js`
- 修改：`public/js/app.js`

**缓存策略**

| 数据类型 | 键 | TTL |
|----------|-----|-----|
| 模板列表 | `all` | 24 小时 |
| 卖点推荐 | `brandModel::year` | 7 天 |
| 车辆信息提取 | `textHash` | 1 小时 |

**客户端调用流程**

```
进入模板页
  → 检查 IndexedDB 缓存
  → 缓存命中？使用缓存数据（无网络请求）
  → 缓存未命中？请求网络 → 存入缓存
```

**验证方案**

1. 缓存命中测试：首次加载模板（网络请求），第二次加载（无网络请求）
2. TTL 测试：修改时钟超过 24h，验证模板缓存失效并重新请求
3. 断网测试：完全断网后进入模板页，验证仍可使用缓存数据

### 优化项 6：图片嵌入阶段优化

**问题描述**

图片嵌入阶段（`embedImagesInSvg`）占用 28%-82% 的进度区间，是用户感知最长的等待段。当前并发数固定为 6，且进度曲线不够平滑。

**优化方案**

1. 进度曲线优化：使用幂函数平滑进度，消除长时间"卡住"的感知
2. 动态并发数：根据设备性能调整（低端机 3，高端机 10）

**涉及文件**

- 修改：`public/js/posterImageEmbedClient.js`
- 修改：`public/js/posterRenderClient.js`
- 修改：`public/js/clientSettings.js`

**进度曲线优化**

```javascript
// 使用幂函数平滑进度
const smoothPercent = 28 + Math.round(54 * Math.pow(done / total, 0.8));
```

**动态并发配置**

```javascript
// clientSettings.js 中新增
const DEFAULTS = {
  embedConcurrencyLowEnd: 3,   // 4核以下
  embedConcurrencyHighEnd: 10, // 8核以上
};

export function getEmbedConcurrency() {
  const isLowEnd = navigator.hardwareConcurrency <= 4;
  return isLowEnd ? getClientSetting('embedConcurrencyLowEnd') : getClientSetting('embedConcurrencyHighEnd');
}
```

**验证方案**

1. 低端机测试（4 核设备）：验证并发数为 3，内存不溢出
2. 高端机测试（8+ 核设备）：验证并发数为 10，嵌入时间缩短
3. 进度平滑测试：验证进度条无倒退、无长时间停顿

## 六、风险评估

### 6.1 本地推荐准确率下降

**风险**：客户端本地卖点推荐依赖模板匹配，不如 LLM 灵活。

**应对**：
- 本地推荐作为首选，LLM 作为 fallback
- 用户仍可通过"重新提炼"按钮触发 LLM 模式
- 持续优化 `localGenerateSellingPoints` 模板覆盖范围

### 6.2 客户端内存占用增加

**风险**：BRAND_META（27 品牌）+ SEED_POINTS（32 条）+ 缓存数据增加内存。

**应对**：
- 总数据量约 5KB（JSON），浏览器内存可忽略
- IndexedDB 缓存设置合理的 maxEntries 限制
- 低端机减少图片嵌入并发数

### 6.3 服务端/客户端逻辑不一致

**风险**：迁移后，服务端和客户端的同一函数行为出现差异。

**应对**：
- 所有迁移函数使用共享的单元测试用例
- 关键函数进行输出比对测试
- 在 CI 中增加"客户端 vs 服务端一致性"检查

### 6.4 浏览器兼容性问题

**风险**：IndexedDB、`AbortController` 在部分旧浏览器不支持。

**应对**：
- 所有 IndexedDB 操作已有 fallback（内存 Map 降级）
- `AbortController` 使用已有 polyfill 模式
- 目标浏览器为现代移动端浏览器（iOS Safari 13+ / Android Chrome 80+）

### 6.5 缓存数据过期

**风险**：卖点模板数据更新后，客户端缓存未刷新。

**应对**：
- 所有缓存设置 TTL（模板 24h，卖点推荐 7d）
- 新增服务端版本号接口，客户端启动时比对
- 管理员后台更新数据时推送版本号变更
